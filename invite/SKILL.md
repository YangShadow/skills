# Android 校招 / 实习生面试指南生成器

## 目标

根据候选人 PDF 简历，为其：
1. 在简历所在目录下创建以姓名命名的子文件夹，将 PDF 移入
2. 结合简历内容和面试模板，生成一份量身定制的面试指南 `姓名-面试指南.md`

---

## 工作流程

### Step 1：定位简历文件和模板

**简历文件定位：**
- 如果用户明确指定了文件路径，直接使用
- 如果用户说"有新简历"但未指定路径，用 `ls` 扫描工作目录及其一级子文件夹，找到未归档的 `.pdf` 文件（即不在姓名子文件夹内的 PDF）
- 确认简历文件后，记录其所在的**父目录**作为工作根目录（下称 `<root>/`）


### Step 2：整理文件结构

对于每一份待处理的 PDF 简历：
- 从文件名中提取候选人姓名（通常是"姓名-简历.pdf"或"姓名_简历.pdf"的格式，取第一个分隔符前的部分）
- 如果文件名无法解析，读取 PDF 内容提取"姓名"字段
- 确认姓名后，在 `<root>/` 下创建 `<root>/姓名/` 子文件夹
- 将 PDF 移入子文件夹

### Step 3：读取简历内容

用 Read 工具读取 PDF 文件，提取以下信息：
- **基本信息**：姓名、学历、院校、专业、毕业时间
- **专业技能**：熟悉 / 熟练 / 了解的技术点，区分掌握程度
- **项目经历**：项目名称、技术栈、主要功能、本人负责内容
- **实习经历**：公司名称、时间、技术方向、主要工作内容
- **荣誉**：竞赛、奖学金等
- **公开链接**：GitHub 主页、技术博客、作品集网站等

### Step 3.5：访问博客和 GitHub（如有）

读取简历内容后，检查是否包含以下类型的公开链接：
- **GitHub**：个人主页（如 `github.com/xxx`）
- **技术博客**：个人博客域名或链接（如 `blog.xxx.cn`、CSDN 主页、掘金主页等）

如果存在，使用 WebFetch 工具访问这些页面，重点关注：

**GitHub 主页访问重点：**
- Pinned 仓库或 star 数较高的仓库：项目名称、主要语言、README 描述
- 最近的提交记录（频率、commit message 质量）
- 代码质量：变量命名、注释习惯、架构组织
- 是否有 Android 相关仓库，挑选 1-2 个具体项目深入了解（访问仓库页面查看 README 和代码结构）

**博客访问重点：**
- 最近发布的文章列表和主题
- 挑选 1-2 篇技术深度较高的文章，了解核心论点和结论
- 文章质量：是否有自己的理解，还是纯粹搬运

**基于访问结果，在面试指南的「二、核心能力考察」末尾额外追加 2 道题**，标注来源：

```markdown
#### 🔗 来源：GitHub / 博客

| 考察领域 | 定制化题目 | 建议题型 | 评分(1-5) |
|---------|----------|---------|----------|
| 【领域】 | 【基于实际内容生成的题目正文】 | 项目追问 | __ / 5 |
| 【领域】 | 【基于实际内容生成的题目正文】 | 技术深挖 | __ / 5 |
```

**出题原则：**
- 题目必须基于实际访问到的内容，不得凭空捏造（如仓库名、文章标题、具体技术决策）
- 优先围绕候选人做过的具体决策或写过的具体内容提问，而非泛泛问"你的项目是做什么的"
  - 例如：「你的 XXX 仓库里用了 YYY 架构，当时为什么选这个方案而不是 ZZZ？」
  - 例如：「你在《博客标题》里提到了 AAA 的优缺点，在你实际项目里是怎么取舍的？」
- 如果 GitHub 仓库代码可访问，可以针对具体实现提问（如某个类的设计、某段逻辑的处理方式）
- 如果链接无法访问（404 / 私有 / 空仓库），跳过此步骤，在简历评估的「代码可见度」项标注 ⚠️ 并写明原因
- 每道题同样需要附「参考答案与评分标准」，参考答案基于访问内容给出预期的优秀回答要点

### Step 4：参考内置面试框架

面试指南框架已内置于本 skill，无需外部模板文件。按以下结构和考察标准生成面试指南。

---

#### 📋 内置框架：简历筛选标准

| 筛选维度 | 具体要求 | 优先级 |
|---------|---------|-------|
| **项目产出** | 有实际 Android 项目（课程项目 / 个人 App 均可）；项目有明确的功能描述和技术栈说明 | 必要 |
| **代码可见度** | GitHub / Gitee 有可查看的代码仓库；有开源贡献经历加分 | 加分 |
| **技术社区参与** | 技术博客、技术分享、竞赛经历等；能体现对技术的主动探索和热情 | 加分 |

---

#### 📋 内置框架：核心能力考察矩阵

**2.1 计算机基础**（校招生基础功底是区分度最大的指标）

| 考察领域 | 具体要求 | 建议题型 |
|---------|---------|---------|
| **数据结构与算法** | 链表、树、图的基本操作；常见排序与搜索；时间/空间复杂度分析 | 口述题（线上无编码环境，要求口述思路、数据结构选择、复杂度分析） |
| **操作系统** | 进程与线程的区别；内存管理基础；进程间通信（关联 Binder / Handler） | 问答题 |
| **网络基础** | HTTP / HTTPS 基础；TCP 三次握手 / 四次挥手；网络请求框架理解 | 问答题 |

**2.2 Kotlin 语言能力**

| 考察点 | 具体要求 | 建议题型 |
|-------|---------|---------|
| **协程 (Coroutines)** | 协程与线程的区别；suspend 函数基本原理；作用域（scope）概念 | 场景题 |
| **语言特性** | 空安全设计；高阶函数 / Lambda；扩展函数 / 数据类 | 场景题 |

**2.3 Android 平台理解**

| 考察点 | 具体要求 | 建议题型 |
|-------|---------|---------|
| **生命周期** | Activity / Fragment 生命周期；配置变更场景；数据保存与恢复 | 场景题 |
| **Jetpack 组件** | ViewModel / LiveData / StateFlow；Room 数据库；Navigation 组件 | 问答题 |
| **Jetpack Compose** | 声明式 UI 基本概念；Composable 函数与状态管理；有实践经验加分 | 场景题 |

**2.4 进阶加分项**

| 考察点 | 具体要求 | 建议题型 |
|-------|---------|---------|
| **架构意识** | MVVM / MVI 基本思想；关注点分离；能解释为何不把逻辑写在 Activity 里 | 设计讨论 |
| **性能意识** | 内存泄漏常见原因；Android Studio Profiler 使用；基本优化思路 | Debug 场景题 |
| **工程实践** | CI/CD 流程了解；单元测试经验；KMP 等跨平台技术了解 | 问答题 |

---

#### 📋 内置框架：AI 能力三层模型

2026 年，AI Coding 能力已从"加分项"变为"基本素养"。考察的本质不是会不会用 AI，而是能不能用好 AI。

| 层级 | 能力名称 | 考察要点 |
|-----|---------|---------|
| **L1** | 判断力 | 知道什么场景该用 AI，什么不该；能识别 AI 的能力边界 |
| **L2** | 鉴别力 | 能审查和修正 AI 生成的代码；能发现典型的 AI 代码问题模式 |
| **L3** | 整合力 | 能把模糊需求转化为有效的 Prompt；能将 AI 产出融入工程实践 |

---

#### 📋 内置框架：建议面试流程

| 轮次 | 主题 | 内容 | 时长 |
|-----|-----|------|-----|
| **第一轮** | 基础编程 | 中等难度算法题 1 道（口述）；Kotlin 语言特性相关题 1 道 | 45 分钟 |
| **第二轮** | Android 专项 | 生命周期场景题；架构设计讨论；AI 能力考察 | 60 分钟 |
| **第三轮** | 综合面 | 项目经历深挖；学习方法与技术热情；团队协作与沟通能力 | 30 分钟 |

---

#### 📋 内置框架：综合评分表结构

| 评估维度 | 权重 |
|---------|-----|
| 计算机基础 | 25% |
| Kotlin 语言能力 | 20% |
| Android 平台理解 | 20% |
| AI 能力（判断 + 鉴别 + 整合） | 20% |
| 学习能力与技术热情 | 15% |
| 进阶加分项（架构/性能/工程） | 加分 |

> **核心筛选原则**：校招生的学习能力和技术热情往往比当前技能水平更重要。一个基础扎实、学东西快、对技术有好奇心的候选人，三个月后的产出可能远超一个只是背了很多八股文的人。考察技术热情的建议问题："你最近学了什么新技术，遇到了什么坑？"

### Step 5：生成专属面试指南

生成 `<root>/姓名/姓名-面试指南.md`，遵循以下规则：

**文件头部**：添加候选人基本信息摘要和简历亮点评述。

**一、简历评估**：对照模板的筛选标准（项目产出 / 代码可见度 / 技术社区参与），评估候选人简历，用 ✅ / ⚠️ / ❌ 标注，并写明理由。

**二、核心能力考察**：这是定制化的核心。对于每个考察点：
- 算法题统一改为**口述题**：不要求写代码，让候选人口述解题思路、选用的数据结构及原因、时间/空间复杂度分析，以及必要时的追问（如"有没有更优方案"）。这适应线上无编码环境的面试形式，同时更能考察候选人的思维清晰度而非语法记忆
- 通用型知识点（如算法基础、HTTP 协议）保留模板题目即可
- 凡是候选人简历中**明确提到的技术**，必须结合其项目经历设计场景化题目
  - 例如：候选人用过 Hilt，就问"你为什么用 Hilt，它帮你解决了什么问题"
  - 例如：候选人用过 Media3 + Service，就用他的场景出生命周期题
  - 例如：候选人有竞赛经历，算法题难度可以适当提高
- 题目要具体、有情境感，避免泛泛而问
- **每道题之后必须紧跟「参考答案与评分标准」**（格式见下方说明）
- 如果 Step 3.5 有访问到 GitHub / 博客内容，在本节末尾追加「🔗 来源：GitHub / 博客」专区（2 道题）

**三、AI 能力考察**：AI 三层模型表格保持通用结构，考察题包含三种题型：
- **题型一：如何审核 AI 生成的代码**（非实操，描述性问答——结合候选人的项目技术栈，问"如果 AI 帮你写了一段 XXX 代码，你在整合进项目前会怎么审核，重点检查哪些地方"，考察审查思路的系统性和对 AI 常见错误模式的了解）
- **题型二：AI 辅助编程思路**（非实操，描述性问答——给出一个贴近候选人项目的编程需求，问"如果你用 AI 辅助完成这个需求，你会怎么做"，考察 Prompt 拆解、输出审查、验证方法）
- **题型三：开放式问题**（结合候选人项目中实际的 AI 使用经历展开）

每道 AI 考察题同样需要附上参考答案与评分标准。

**四、建议面试流程**：结合候选人技术栈亮点，调整三轮面试的侧重点描述，但三轮结构保持不变。

**五、核心筛选原则**：在模板原则基础上，加入针对该候选人的具体建议（如"重点确认 GitHub 是否可见"、"需确认研究生实习时间"等）。

**六、综合评分表**：保持模板格式，填入候选人姓名，加分项描述可结合候选人的具体技术背景微调。

## 参考答案与评分标准的写法

每道题（包括编程题、问答题、场景题）的表格行之后，紧跟一个引用块，格式如下：

```markdown
> **📋 参考答案与评分标准**
>
> **参考答案**
>
> （写出该题的标准答案或答题要点，编程题给出完整代码 + 关键注释，场景题列出核心思路）
>
> **评分标准（1-5 分）**
>
> | 分数 | 标准 |
> |-----|------|
> | 5 | （优秀：答出所有要点，有延伸思考或追问能回答） |
> | 4 | （良好：答出核心要点，细节稍有欠缺） |
> | 3 | （一般：答出主要思路但不完整，或有明显错误但能自我纠正） |
> | 2 | （较差：答出部分但遗漏关键点，追问也无法补全） |
> | 1 | （不合格：基本没有思路，或答案完全错误） |
>
> **追问方向**（可选，适合有深度的题目）
>
> - （追问 1）
> - （追问 2）
```

**评分标准写作要求：**

- 5 分标准要明确说明"优秀答案"包含哪些具体要点，不要只写"答得很好"
- 每档分数的边界要清晰，让不同面试官打分时结论趋于一致
- 编程题的评分还应考虑：代码是否能编译运行、边界条件是否考虑、时间复杂度是否最优
- 场景题的评分应考虑：是否结合自身项目经验、是否有实际操作过还是只停留在理论层面
- 追问方向帮助面试官在候选人回答不完整时引导深入，或在候选人表现优秀时进一步拉开差距

---

## 质量要求

- 题目要有**情境感**，不是泛泛的"说说 XXX 是什么"，而是"你在 YYY 项目里用了 XXX，当时遇到了什么问题，你是怎么处理的"
- **每道题必须附参考答案与评分标准**，这是生成指南的必要组成部分，不可省略
- 保持与模板相同的 Markdown 格式和表格结构，方便直接打印使用
- 评分列（__ / 5）保持空白，供面试官现场填写
- 如果候选人简历技术栈和模板考察点高度匹配，部分题目可以直接用模板题目，不必强行改写

```
    └── 姓名-面试指南.md        ← 新生成
```
