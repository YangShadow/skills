# prd-writer 使用指南

## 是什么

`prd-writer` 是一个 Claude Code Skill，用于辅助产品经理撰写、审查和维护产品需求文档（PRD）。它内置了文档模板、边界场景扫描、逻辑一致性检查、数值测算等能力，能将一段粗糙的产品描述转化为研发可直接评审的结构化文档。

---

## 快速启动

在 Claude Code 中直接说：

```
帮我用 prd-writer 写一份 XXX 的需求文档
```

或者：

```
用 prd-writer 审查这份 PRD：[粘贴文档]
```

Claude 会自动调用该 Skill 并进入对应模式。

---

## 支持的模式

| 模式 | 触发场景 | 输入 | 输出 |
|------|---------|------|------|
| **Draft（起草）** | 从零写一份新 PRD | 一段粗糙描述、几条 bullet | 完整结构化 PRD |
| **Review（审查）** | 检查已有 PRD 的质量 | 现有 PRD 文本 | 逻辑漏洞报告 + 健康评分（0-10） |
| **Reconcile（对齐）** | PRD 与交互稿出现差异 | PRD + 交互说明/Figma 批注 | 差异对照表 + 更新段落 |
| **Interrogate（压测）** | 上评审前想提前挑刺 | 确认后的 PRD | AI 扮演研发+测试逐章追问 |
| **Split（拆分）** | 大 PRD 需要按岗位分发 | 完整 PRD | 按角色生成子文档 |

## Draft 模式详解（最常用）

### 支持的文档类型

| 类型 | 适用场景 |
|------|---------|
| A — 直播间玩法/小游戏 | 直播间内的互动活动 |
| B — 运营活动 | 限时促销、节日活动 |
| C — 会员权益 | 会员购买、权益设计、促活 |
| D — 微信小游戏 | 微信平台小程序游戏 |
| E — 其他 | 不在以上分类内的功能 |
| F — 通知/提示/弹窗 | 弹窗文案、Push 规则 |
| G — 埋点需求 | 数据埋点设计文档 |
| H — 运营告知书/宣发 | 给运营和市场的执行说明 |
| I — 独立游戏 | 消除类等独立部署游戏 |
| J — 语音歌房 | 多人实时语音互动房间 |
| K — K 歌录音棚 | 录歌、混音、发布功能 |

### 自动执行的流程

告诉 Claude 你的需求后，Draft 模式会**自动**完成以下步骤，无需逐一提示：

```
① 读取对应类型模板
② 生成完整草稿（每章标注 ✅ / ⚠️ / ❌）
③ 自动扫描 8 个维度的边界场景漏洞
④ 自动检查逻辑顺畅性
⑤ 主动询问配套模块（埋点/消息通知/宣发告知）
```

后续步骤按需触发：可视化流程图、数值测算、Dev Checklist、通用规范等。

### 置信度标记含义

| 标记 | 含义 | 你需要做什么 |
|------|------|------------|
| ✅ | 信息充足，逻辑自洽 | 快速确认即可 |
| ⚠️ | 信息模糊，Claude 列出 2-3 个选项 | 从选项中选一个，或提供自己的方案 |
| ❌ | 信息严重不足，无法填写 | 必须补充信息后 Claude 重新填写 |

---

## 配套模块（Layer 4）

草稿完成后，Claude 会主动询问以下配套模块是否需要生成，**不需要你主动提**：

- **埋点需求**：事件列表、参数字段定义、看板需求、验收标准
- **消息通知**：Push / 弹窗 / 站内信的触发时机、频次、降级规则
- **宣发与运营告知**：后台配置项清单、宣发文案方向、FAQ、应急预案
- **按岗位拆分**：美术需求单、研发评审文档、运营告知书、埋点文档

回复「需要」或直接说「继续」即可全部生成。

---

## Review 模式：健康评分标准

| 分数 | 结论 | 建议行动 |
|------|------|---------|
| 8–10 | 文档完整，可直接评审 | 按建议微调后提交 |
| 6–7 | 局部缺失，修补后可评审 | 按复检报告逐项补充 |
| < 6 | 结构性问题 | 建议从头重新梳理，不要继续修补 |

## 典型工作流示例

### 场景 1：从零起草

```
我：帮我用 prd-writer 写一份需求文档。
   需求是：购买音乐会员赠送999足金锦鲤抽奖机会，月卡送7次，
   连续72次未中保底赠实物。

Claude：（自动识别为 C 类会员权益文档，生成草稿 + 边界扫描 + 询问配套模块）

我：继续（生成全套配套文档）
```

### 场景 2：审查已有文档

```
我：用 prd-writer review 模式审查这份文档：[粘贴 PRD]

Claude：（输出逻辑冲突列表、边界漏洞、模糊表述、健康评分）
```

### 场景 3：交互稿对齐

```
我：PRD 和交互稿有出入，帮我用 reconcile 模式对齐：
   [粘贴 PRD] [粘贴交互说明]

Claude：（输出差异对照表 + 建议以哪份为准 + 更新后的段落）
```

## 常见问题

**Q：我不知道属于哪种文档类型，怎么办？**  
直接描述需求，Claude 会自动判断类型；如有疑问会询问你确认。

**Q：草稿里有很多 ⚠️ 需要我回答，能不能一次性全答完？**  
可以，直接把所有问题的答案整段发过来，Claude 会逐条解析并更新文档。

**Q：生成了很多 ❌，说明我的描述不够详细吗？**  
是的。❌ 表示该信息必须由你提供，Claude 无法推断。补充后可以让 Claude 重新填充该章节。

**Q：文档版本怎么管理？**  
每次重大修改后 Claude 会更新文档头部的版本号（v0.1 → v0.2 → ...），建议在 Git 中对文件做版本控制。

**Q：Interrogate 模式会很烦吗？会一直追问吗？**  
Interrogate 模式会逐章提问，每章 2-4 个问题，你可以随时说「跳过这章」或「结束追问」。适合在正式评审前自测用。
